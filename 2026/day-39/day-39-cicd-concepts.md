# Day 39 – What is CI/CD?

## Task 1: The Problem

### Scenario

A team of 5 developers is working on the same application. Everyone pushes code to the same repository and deployments are done manually.

### What can go wrong?

* Developers may overwrite each other's code.
* Bugs can reach production without testing.
* Different versions of code may be deployed accidentally.
* Human deployment mistakes can cause downtime.
* Rollbacks become difficult.
* Configuration differences between environments can cause failures.

### What does "It Works on My Machine" mean?

"It Works on My Machine" means an application runs successfully on a developer's computer but fails in another environment such as testing, staging, or production.

Reasons:

* Different software versions
* Missing dependencies
* Different operating systems
* Different environment variables
* Configuration mismatches

This is a real problem because users never run the application on the developer's machine—they run it in production.

### How many times a day can a team safely deploy manually?

Usually only a few deployments per day can be performed safely.

As deployment frequency increases:

* Human errors increase
* Deployment time increases
* Rollback complexity increases
* Production risks increase

This is why automation through CI/CD is important.

# Task 2: CI vs CD

## Continuous Integration (CI)

Continuous Integration is the practice of automatically building and testing code whenever developers push changes to a repository.

CI helps detect bugs, integration issues, and build failures early.

### Example

A developer pushes code to GitHub.

GitHub Actions automatically:

* Builds the application
* Runs unit tests
* Reports success or failure

---

## Continuous Delivery (CD)

Continuous Delivery extends CI by automatically preparing applications for release after successful testing.

The deployment is ready but requires manual approval before going to production.

### Example

After tests pass:

* Docker image is created
* Application is deployed to staging
* Team approves production release manually

---

## Continuous Deployment

Continuous Deployment automatically deploys every successful change to production without manual approval.

If all tests pass, deployment happens automatically.

### Example

A company like Netflix can automatically deploy tested changes directly to production several times per day.

---

# CI vs Delivery vs Deployment

| Feature                         | Continuous Integration | Continuous Delivery | Continuous Deployment |
| ------------------------------- | ---------------------- | ------------------- | --------------------- |
| Build Code                      | Yes                    | Yes                 | Yes                   |
| Run Tests                       | Yes                    | Yes                 | Yes                   |
| Create Release                  | No                     | Yes                 | Yes                   |
| Manual Approval Needed          | N/A                    | Yes                 | No                    |
| Automatic Production Deployment | No                     | No                  | Yes                   |

# Task 3: Pipeline Anatomy

## Trigger

The event that starts a pipeline.

Examples:

* Git push
* Pull Request
* Scheduled execution
* Manual execution

---

## Stage

A logical section of a pipeline.

Examples:

* Build
* Test
* Deploy

---

## Job

A group of related tasks executed within a stage.

Example:

Build Stage

Job:

* Build Docker image

---

## Step

A single action or command inside a job.

Example:

```bash
docker build -t myapp .
```

---

## Runner

The machine that executes pipeline jobs.

Examples:

* GitHub-hosted runner
* Jenkins agent
* GitLab runner
* Self-hosted runner

---

## Artifact

A file or package generated by a pipeline.

Examples:

* Docker image
* JAR file
* ZIP package
* Test reports

# Task 5: Explore in the Wild

## Repository Chosen

## Task 5: Explore in the Wild

### Repository Chosen

React

Repository: https://github.com/facebook/react

### Workflow File

```text
.github/workflows/runtime_build_and_test.yml
```

### What Triggers It?

* Push events
* Pull Request events

### How Many Jobs Does It Have?

The workflow contains multiple jobs, including:

* Build
* Test

### What Does It Do?

This workflow helps ensure that new code changes do not break React.

Main activities include:

* Checking out the source code
* Installing dependencies
* Building React packages
* Running automated tests
* Verifying that the application works correctly

### Why Is It Important?

This workflow helps maintain code quality by automatically validating code changes before they are merged into the main branch.

Benefits:

* Detects bugs early
* Prevents broken code from being merged
* Ensures successful builds
* Maintains software reliability

### My Understanding

React uses GitHub Actions workflows to automate building and testing. Whenever a developer pushes code or creates a pull request, the workflow runs automatically to verify that the changes are safe and do not introduce errors.

# Key Learnings

* CI/CD is a development practice, not a tool.
* CI finds problems early through automated testing.
* Continuous Delivery prepares software for release.
* Continuous Deployment releases software automatically.
* Pipelines automate repetitive tasks.
* A failed pipeline prevents bad code from reaching production.
* Automation improves reliability, speed, and consistency.
